# 23 · Security

Defence-in-depth specifically for an on-premise multi-tenant industrial platform. Goes beyond auth (covered in [14_AUTH_KEYCLOAK](14_AUTH_KEYCLOAK.md)).

## 1 · Threat model summary

| Threat | Source | Primary control |
|---|---|---|
| Stolen operator credentials | Phishing, insider | MFA + short token TTL + audit |
| Lateral movement after pod compromise | Worm / supply chain | Network policy, least-privilege DB roles |
| Tampered telemetry | Compromised edge | Quality envelope + signed agent commands |
| Unauthorised write-back to equipment | Compromised agent / user | Approval gates Tier 3+, dual auth Tier 5 |
| Data exfiltration | Insider | Audit log + egress allowlist |
| LLM prompt injection via knowledge corpus | Malicious doc | Validator agent + corpus signing |
| Ransomware | External | Offline backup + PITR restore drill |
| Cross-tenant data leak | Bug | Row-Level Security + tenant-scoped tokens |

## 2 · Data at rest

| Asset | Encryption |
|---|---|
| Postgres data volume | LUKS at the disk layer (Linux production); Docker volumes on encrypted filesystem in POC |
| Postgres column-level (`app.api_keys.hash`, `app.users.email`) | Application-side `pgcrypto` for any PII |
| Kafka log segments | LUKS at disk layer; topic ACLs prevent broker-side cross-service reads |
| MinIO buckets (reports, backups) | SSE-S3 server-side encryption with a customer-supplied key |
| Audit table | Same as Postgres; additionally `INSERT`-only role prevents tampering even with DB compromise |

## 3 · Data in transit

| Edge | TLS source |
|---|---|
| Browser → frontend | nginx with Let's Encrypt or customer-CA cert |
| Frontend → api-service / ws-bridge | Same TLS edge (proxy in front) |
| Service-to-service inside compose | PLAINTEXT in POC (single host) → mTLS in v2 production |
| api-service → Anthropic | TLS 1.3, certificate-pinned in production |
| api-service → Keycloak | TLS, JWKS endpoint cached |
| dal-bacnet → Kafka | PLAINTEXT (LAN) in POC → SASL_SCRAM + mTLS in production |
| Backups to MinIO | TLS |

## 4 · Secrets

- POC: `.env` file, **gitignored**, generated by `make .env` from prompts.
- Production: External Secrets Operator → Vault (or AWS Secrets Manager if hybrid).
- Anthropic API key rotated quarterly; key per environment.
- DB role passwords rotated every 90 d; documented runbook in `infra/operations/SECRETS_ROTATION.md`.
- No secret ever logged. Pino redaction list configured for every service.

## 5 · Network

| Plane | Allowed |
|---|---|
| Edge inbound | 80, 443 only (Traefik/nginx); 8765 only via the same TLS edge |
| Inter-service | Docker `omnyx_net` only |
| Database | 5432 not exposed beyond compose network |
| Kafka | 9092 not exposed; UI on 8080 LAN-only |
| Egress | api-service: only `api.anthropic.com`. agentic-ai: only `api.anthropic.com` and `ollama` host. All others: deny. |
| BACnet UDP | Host networking for `dal-bacnet`; ACL on the building network limits the listening port |

## 6 · Authentication & authorisation

- Detailed in [14_AUTH_KEYCLOAK](14_AUTH_KEYCLOAK.md).
- Reinforcement here:
  - **No long-lived tokens** — access 5 min, refresh 30 min.
  - **Approval tokens** for Tier 3+ tool calls — 15-min single-use, signed by api-service.
  - **Service accounts** for inter-service calls; each service has its own client with the minimum role required.
  - **RBAC + ABAC** — RBAC for "what kind of action" (read alerts vs ack alerts), ABAC for "which tenant/site this user can see".

## 7 · LLM / Agent specific

- **Validator agent** is the second line on tool calls — independent context.
- **Loop guard** aborts after 5 same-tool calls / workflow.
- **Token budget** per workflow + monthly customer cap; alerts fire at 80 %.
- **Prompt cache** on system prompt + tool defs — reduces cost, doesn't expose user data because user-specific content goes after the cache breakpoint.
- **Knowledge corpus signing** — every ingested doc has a SHA-256 + author and the agents are instructed (system prompt) to refuse instructions inside knowledge documents.
- **Tool input/output schema validation** with Zod — agents can't pass arbitrary JSON.
- **No agent talks to the database directly** — all writes go through api-service tool gateway which enforces RBAC and approval gates.

## 8 · Audit & compliance

- `audit.events` is append-only at the DB role level (POC) and append-only-with-immutable-replication in production.
- Every write-back to equipment (`write_setpoint`, `change_mode`, `start_equipment`, `stop_equipment`, `plant_shutdown`) generates an audit row with actor (user or agent), payload, approval id.
- 90-day audit retention for live queries, 7-year cold archive in MinIO (configurable per customer).
- Compliance modes (GDPR / SOX / HIPAA-equivalent) are feature-flagged; specific customers can enable stricter retention/encryption per `app.feature_flags`.

## 9 · Vulnerability & supply-chain

- Trivy scan on every image build in CI ([26_CI_CD](26_CI_CD.md)).
- Dependency policy: critical CVE → patched within 7 d.
- SBOM (CycloneDX) generated per image.
- Image signatures verified at pull (cosign in production).
- Base images: pinned by SHA, not tag.

## 10 · Incident response

- Runbook at `infra/operations/INCIDENT_RESPONSE.md` (placeholder; written during W12 hardening).
- On security event:
  1. Pause all agentic-ai workflows (`POST /admin/agents/pause`).
  2. Force token refresh (rotate Keycloak signing keys).
  3. Snapshot audit table + Kafka logs to cold storage.
  4. Review per the runbook.

## 11 · Penetration test plan

- POC: internal review of OWASP Top 10 against api-service routes.
- Beta: external pentest by reputable firm before first beta customer.
- v2: continuous (Burp + nuclei) in CI.

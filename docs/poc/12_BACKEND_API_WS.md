# 12 · Backend (api-service) + WebSocket bridge

Fastify + TypeScript service. Two responsibilities:
1. REST + WebSocket surface for the React UI.
2. **Tool gateway** for agentic AI (every tool the agents can call is a Fastify route here — see [11_AGENTIC_AI.md §3](11_AGENTIC_AI.md)).

The `ws-bridge` is a separate small service (Node) because it has a different scaling profile (one connection per logged-in client) and because we want a clear failure boundary between REST and live push.

## 3 · Rules engine

Lightweight evaluator running in-process. On every `PointBatch` from Kafka `raw.bacnet.*`:

```
for rule in active_rules_for_point(point_id):
    if rule.type == "threshold":     evaluate threshold + hold-time
    if rule.type == "offline":       evaluate consecutive misses
    if rule.type == "anomaly":       z-score over rolling baseline
    if rule.type == "delta":         rate-of-change rule
  → if fires, create Alert via repository + emit Kafka
```

Rules are JSON in `app.rules` (table not detailed in 08 because it follows the alerts shape — a thin `point_id, kind, params JSONB, severity, hold_seconds` row). DQ flags interact per the matrix in [06_DATA_QUALITY_LAYER.md §4](06_DATA_QUALITY_LAYER.md).

## 4 · Scheduler

BullMQ on Redis. Jobs:
- `pm_task_runner` — generates work orders for upcoming PM intervals.
- `daily_report_kick` — triggers Agentic AI Workflow B at 06:00.
- `rl_review_weekly` — triggers RL review workflow on Sundays.
- `tier2_etl_kick` — fires a Kafka event the `dq-etl` container listens for (the actual ETL still runs in dq-etl; the kick is just to keep the schedule centralised).

## 5 · WebSocket bridge (`ws-bridge`)

Small service, separate process. Subscribes to Kafka `raw.bacnet.*`, `agent.activity`, `twin.fdd.alerts`, and rebroadcasts to clients.

```
Client connects → wss://host:8765/?token=<JWT>
  → ws-bridge verifies JWT with Keycloak public key
  → subscribes the client to relevant channels based on RBAC
       'plant.snapshot.{site_id}'   — every 5 s aggregated PlantSnapshot
       'alerts.{site_id}'           — fire-and-forget on new alert
       'agent.activity.{workflow_id}' — only when subscribed (UI opts in)
  → broadcasts at < 5 s latency
```

PlantSnapshot is built by an in-memory rolling state inside `ws-bridge`:

```
on every PointBatch:
  for reading in batch:
    state[site][device].metrics[point_id] = MetricLatest(...)
every 5s:
  emit PlantSnapshot to subscribed clients
```

Sized per gl_pbs benchmarks: ~10 msg/s in our load, < 5 % CPU; same WebSocket model already proven in `plant_snapshot_api.py`.

## 6 · Auth & RBAC

- Every route guarded by `@fastify/jwt` verifying Keycloak-issued JWTs.
- `req.user.roles` populates from JWT claims.
- Decorator `requireRole('operator','manager')` per route.
- Tool gateway has an extra `requireApprovalTier(n)` decorator — Tier 3+ tools must come with an approval token, or a Tier-3-trusted agent's identity, depending on workflow config.

See [14_AUTH_KEYCLOAK.md](14_AUTH_KEYCLOAK.md).

## 7 · Observability

- `pino` JSON logs → Loki via Promtail.
- `prom-client` metrics — `http_request_duration_seconds`, `kafka_consume_lag`, `rule_eval_total`, `agent_tool_call_total{tool,tier}`.
- `request-id` propagated through every layer for trace.
- Health endpoint `/healthz` reports: Postgres, Kafka, Redis, Keycloak, twin-broker, rl-broker, agentic-ai, llm backend.
